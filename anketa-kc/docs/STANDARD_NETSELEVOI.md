# Стандарт «Целевой / Нецелевой» КЦ

> Свод правил для классификации лида по результатам разговора колл-центра. Лежит в основе автоматического расчёта `target-status.js` в анкете КЦ.
> Источник: внутренний PDF «Стандарт нецелевой встречи» (актуально на 27.04.2026).

---

## 0. Зачем это нужно

Колл-центр (КЦ) собирает анкету по клиенту и должен передать менеджеру по продажам (МП) только лиды, по которым процедура банкротства физлиц действительно поможет. Если очевидных стоп-факторов много — встречу всё равно можно назначить (мягкое предупреждение), но в БП «Назначить встречу» уйдёт статус «Нецелевой», МП увидит причины в таймлайне и подготовится иначе.

Статус хранится в поле лида `UF_CRM_1649136704` (enum: 289 = Целевой, 290 = Нецелевой, 291 = Не определено) и передаётся в БП 40 «Назначить встречу» как обязательный параметр `CelNeCel`.

---

## 1. Алгоритм определения статуса

```
formData (поля анкеты + 14 чек-боксов признаков)
   ↓
TargetStatus.evaluate(formData)
   ↓
{ id: 289 | 290 | 291, label, reasons[] }
```

Базовая логика:

1. Прогнать все правила (`RULES`, 11 штук) по `formData`.
2. Если хотя бы одно правило сработало — статус = **Нецелевой** (290), `reasons` = labels всех сработавших правил.
3. Если правил не сработало и ключевые поля анкеты заполнены — статус = **Целевой** (289).
4. Если ключевые поля пустые / неизвестно — статус = **Не определено** (291).

«Не определено» — мягкое предупреждение: при бронировании КЦ увидит `confirm("«Целевой/Нецелевой» не определён. Записать всё равно?")`. Запись не блокируется.

## 2. Правила (11 штук)

Все правила реализованы в `anketa-kc/assets/target-status.js` в массиве `RULES`. Если правило срабатывает, его `label` попадает в `reasons[]` и в комментарий таймлайна.

### 2.1 Долг менее 300 000 ₽

- `id`: `debt_below_300k`
- **Условие**: `0 < debtTotal < 300000`
- **Логика**: с долгом до 300 000 ₽ выгоднее погасить мирно, банкротство не оправдано.

### 2.2 Залог: клиент не готов с ним расстаться

- `id`: `collateral_not_ready`
- **Условие**: `collateral === 'Y'` И `mortgage !== 'Y'` (это не ипотечный залог) И `collateralReadyToPart === 'N'`
- **Логика**: залог при банкротстве уходит к залогодержателю. Если клиент не готов — банкротство ему не подходит.

### 2.3 Лишнее имущество: переоценено или нет готовности к рискам

- `id`: `extra_property_overprice_or_no_risks`
- **Условие**: `property === 'Y'` И (`propertyOverDebt === 'Y'` ИЛИ `propertyReadyForRisks === 'N'`)
- **Логика**: если стоимость лишнего имущества выше суммы долга — финуправляющий продаст его в счёт долга. Если клиент не готов к таким рискам — банкротство ему не подходит.

### 2.4 Сделки в период просрочки

- `id`: `deals_during_overdue`
- **Условие**: `deals === 'Y'` И `dealsDuringOverdue === 'Y'`
- **Логика**: сделки за 3 года (продажа/дарение) во время просрочки могут быть оспорены финуправляющим как преднамеренный вывод активов.

### 2.5 Банкротство «за другого человека»

- `id`: `for_other_person`
- **Условие**: `forOther === 'Y'`
- **Логика**: процедура назначается лично, нельзя «прислать вместо себя». Если клиент действует за другого — отказ.

### 2.6 ООО с балансом и неготовность расстаться

- `id`: `ooo_with_balance_and_not_ready`
- **Условие**: `ooo === 'Y'` И `oooHasBalance === 'Y'` И `oooReadyToPart === 'N'`
- **Логика**: доля в ООО с активами на балансе — имущество, которое продадут на торгах. Если клиент не готов — отказ.

### 2.7 Долг, не подлежащий списанию

- `id`: `non_dischargeable_debt`
- **Условие**: `nonDischargeable === 'Y'`
- **Логика**: алименты, возмещение вреда жизни/здоровью, субсидиарка, зарплата — не списываются по 127-ФЗ. Если основной долг такой — банкротство бессмысленно.

### 2.8 Другая компания уже подала в АС

- `id`: `other_company_in_arbitration`
- **Условие**: `otherCompanyAS === 'Y'`
- **Логика**: если процедура уже инициирована другой юридической компанией — лид не наш, ведут другие.

### 2.9 Высокий официальный доход

- `id`: `high_official_income`
- **Условие**: `incomeOfficial === 'high'` ИЛИ `incomeKmBad === 'Y'`
- **Логика**: при высоком официальном доходе суд может решить, что должник способен погасить долги в реструктуризацию, а не списать. `incomeKmBad === 'Y'` — флаг «доход вне критериев КМ» (плохой по другим причинам, чем зарплата).

### 2.10 Уголовное дело по ст. 159 УК РФ по тем же основаниям

- `id`: `criminal_159_same_grounds`
- **Условие**: `criminal === 'Y'` И `criminal159SameGrounds === 'Y'`
- **Логика**: если есть приговор/возбуждённое дело по 159-й (мошенничество) по тем же долгам, которые хотят списать — суд откажет в освобождении от обязательств.

### 2.11 Ипотека высокого риска

- `id`: `mortgage_high_risk`
- **Условие**: `mortgage === 'Y'` И (`mortgageNoGuarantor === 'Y'` ИЛИ `mortgageBadOverdue === 'Y'`)
- **Логика**: ипотеку при банкротстве можно сохранить, но только если есть поручитель (или родственник, готовый платить) и нет грубой просрочки. Если ни того, ни другого — квартиру продадут.

## 3. 14 чек-боксов в анкете

Чек-боксы рендерятся в блоке 5 «Признаки нецелевой» в `index.php`. Состояние хранится только в памяти страницы (`formData`), в `crm.lead.update` НЕ отправляется. Передаётся только итоговый статус — в БП «Назначить встречу».

| Ключ в `formData` | Метка в UI |
|---|---|
| `mortgage` | Есть ипотека |
| `mortgageNoGuarantor` | Ипотека: нет поручителя/созаёмщика |
| `mortgageBadOverdue` | Ипотека: грубая просрочка |
| `collateralReadyToPart` | Готов расстаться с залогом |
| `propertyOverDebt` | Стоимость имущества выше долга |
| `propertyReadyForRisks` | Готов к рискам по имуществу |
| `dealsDuringOverdue` | Сделки в период просрочки |
| `oooHasBalance` | У ООО есть баланс |
| `oooReadyToPart` | Готов расстаться с долей в ООО |
| `criminal159SameGrounds` | Ст. 159 УК РФ по тем же основаниям |
| `forOther` | Банкротство «за другого» |
| `nonDischargeable` | Долг, не подлежащий списанию |
| `otherCompanyAS` | Уже подан в АС другой компанией |
| `incomeKmBad` | Доход вне критериев КМ |

Каждый чек-бокс возвращает строку `'Y'` или `'N'` в `formData`.

## 4. Виджет-индикатор

В шапке блока 5 справа от заголовка — бейдж `#target-status-badge`, цвет которого отражает текущий статус:

| Статус | Цвет бейджа (Tailwind) | Текст |
|---|---|---|
| Целевой | `bg-green-100 text-green-800` | Целевой |
| Нецелевой | `bg-red-100 text-red-800` | Нецелевой |
| Не определено | `bg-yellow-100 text-yellow-800` | Не определено |

Под чек-боксами (`#target-status-reasons`) — раскрывающийся блок с маркированным списком причин (`<ul>`). Показывается только когда статус = «Нецелевой». Управляется form.js через CSS-класс `hidden`.

Виджет обновляется в реальном времени: form.js вешает делегированные `change`/`input`-listener'ы на `#anketa-form` и вызывает `updateTargetStatusWidget()` при любом изменении.

## 5. Передача в БП «Назначить встречу»

При бронировании слота в правой панели расписания (`calendar.js → bookSlot()`):

1. Вызывается `updateTargetStatusWidget()` (или fallback `TargetStatus.evaluate(collectFormData())`).
2. Читается `window.__targetStatus = { id, label, reasons[] }`.
3. Если `id === 291` (Не определено) — браузерный `confirm()`. Отказ — выход из бронирования.
4. В `BX24.callMethod('bizproc.workflow.start', PARAMETERS)` добавляется `'CelNeCel': targetStatus.id` (число 289 / 290 / 291).
5. В таймлайн-комментарий добавляется блок:

   ```
   Целевой/Нецелевой КЦ: НЕЦЕЛЕВОЙ. Причины:
   • Долг менее 300 000 ₽
   • Сделки в период просрочки
   ```

   (если статус = Нецелевой; для Целевого — просто строка статуса; для «Не определено» — то же).

## 6. Что НЕ делает приложение

- **Не создаёт новые поля на портале**. Все 14 чек-боксов — UI-only, состояние теряется при перезагрузке (если не сохранено в `crm.lead.update` — а сохранять их в `crm.lead.update` НЕЛЬЗЯ без отдельного согласования).
- **Не блокирует бронирование** при «Не определено». Только мягкое предупреждение `confirm()`.
- **Не пересчитывает существующие лиды** автоматически — статус ставится только при бронировании из анкеты.

## 8. Как изменить правила

1. Открыть `anketa-kc/assets/target-status.js`.
2. Добавить/отредактировать запись в массиве `RULES` (объект `{ id, label, predicate }`).
3. Если правилу нужен новый признак — добавить чек-бокс:
   - В `form.js → initForm()`, блок «netselevoi» — рендер чек-бокса через `fieldCheckbox()`.
   - В `form.js → collectFormData()` — добавить ключ `vc('id_чек-бокса')`.
4. Никаких изменений на портале (Битрикс24) НЕ требуется. Поля анкеты — UI-only.

При добавлении/изменении правил тестировать на лиде 60471 в симуляторе:
`http://localhost:8877/.../bitrix24-local-app-questionnaire/anketa-kc/index.php?leadId=60471`.
