# Техническое задание — Анкета (МКЦ + МП)

> **Статус:** актуальный документ.

---

## Проект

Локальное приложение Bitrix24 для заполнения анкеты клиента. Используется двумя ролями:
- **МКЦ** (менеджер колл-центра) — первичный сбор данных по телефону
- **МП** (менеджер продаж) — уточнение данных и запись клиента на встречу

Открывается как вкладка (таб) в карточке лида CRM. Интерфейс — iframe поверх Bitrix24 REST API и BX24 JS SDK. Backend — PHP.

---

## Утверждённый технологический стек

| Направление | Решение | Примечание |
|---|---|---|
| Frontend styles | **Tailwind CSS 4** | CDN-сборка (`@tailwindcss/browser@4` via jsdelivr), utility-first |
| UI components | **Flowbite 2** | Готовые компоненты поверх Tailwind |
| CRM / portal | **Bitrix24 REST API + BX24 JS SDK** | Локальное приложение, `CRM_LEAD_DETAIL_TAB` |
| Backend | **PHP** | `config.php`, `install.php`, `proxy.php`, `uninstall.php` |

**Правило:** стек зафиксирован. При каждой новой итерации UI не проектируется заново — интерфейс развивается внутри одного утверждённого стека и набора компонентов.

## Требования к frontend

### Общие правила

- Любой новый экран или переработка строится на **Tailwind CSS 4**.
- Стандартные UI-элементы (кнопки, формы, карточки, вкладки, алерты, спиннеры) берутся из **Flowbite**.
- Самописные CSS-файлы (`tokens.css`, `style.css`) **не подключаются** в `index.php` — помечены как устаревшие, оставлены только в git-истории.
- Кастомные `<style>` допустимы только точечно для Bitrix24 iframe/embed-особенностей (reset отступов, кастомный скроллбар).
- Новый дизайн **не проектируется с нуля** при каждой итерации.

### CDN-подключение

```html
<!-- Tailwind CSS v4 CDN -->
<script src="https://cdn.jsdelivr.net/npm/@tailwindcss/browser@4"></script>

<!-- Flowbite 2 CSS -->
<link rel="stylesheet" href="https://cdnjs.cloudflare.com/ajax/libs/flowbite/2.3.0/flowbite.min.css" />

<!-- Flowbite 2 JS (в конце body) -->
<script src="https://cdnjs.cloudflare.com/ajax/libs/flowbite/2.3.0/flowbite.min.js"></script>
```

### Layout

Основной экран — двухколоночный:
- **Левая колонка (55%)** — форма анкеты по блокам с прогресс-баром.
- **Правая колонка (45%)** — расписание МП + вкладка «Фишки» для будущих инструментов.

### Шапка приложения

- Название приложения без указания роли (не «Анкета КЦ» — приложение используется и МКЦ, и МП).
- В правом верхнем углу — **имя текущего пользователя** Bitrix24 (полученное через `BX24.callMethod('user.current')` при инициализации), независимо от роли (МКЦ или МП).

### Блоки формы (левая колонка)

| # | Блок | Поля |
|---|---|---|
| 1 | Финансовые данные | ФИО (auto), сумма долга, платёж, офиц. доход, неофиц. доход, зарплатная карта |
| 2 | Кредитная история | Кредиторы, залог, просрочки, ФССП, имущество, сделки |
| 3 | Личные данные | Место работы, семейное положение, дети, совм. имущество, судимости, ООО, ИП |
| 4 | Заметки менеджера | Исключение из КМ, основная боль, возражения, доп. комментарий |
| 5 | Запись к специалисту | Город клиента (timezone), МП, время клиента, время МП, ID события |

## Требования к расписанию (правая колонка)

### Календари МП

Календари **не привязаны к пользователю Bitrix24** — у каждого МП есть отдельный ID календаря. Слоты получаются через `calendar.accessibility.get` по этому ID, а не по `userId`.

Рабочее время берётся из полей `Время работы ОТ` / `Время работы ДО` каждого календаря (хранятся в Bitrix24, не хардкодятся).

### Слоты

- Шаг слота — **60 минут** (только часовые слоты, получасовых нет).
- Рабочее время МП берётся из полей `Время работы ОТ` / `Время работы ДО` каждого календаря.
- Занятые слоты определяются через `calendar.accessibility.get` по ID календаря и диапазону дат.

### Логика часового пояса

Приоритет — **часовой пояс клиента**.

**Шаг 1 — определение TZ клиента:**
- В блоке 5 МКЦ/МП выбирает город клиента из выпадающего списка (~30 городов России).
- Система определяет UTC-offset клиента из встроенного словаря (не внешний API).

**Шаг 2 — отображение слотов:**
- Каждый слот показывается в **двух колонках**:
  - `Время МП` — локальное время МП (его UTC)
  - `Время клиента` — конвертированное время в TZ клиента
- МКЦ ориентируется на колонку **«Время клиента»** при общении с клиентом.

**Шаг 3 — пересечение рабочего времени:**
- Отображаются только слоты, которые **одновременно** попадают в рабочее время МП и в «разумное» время клиента (09:00–20:00 по TZ клиента).
- Если пересечений сегодня < 3 — автоматически подгружается **следующий рабочий день** (пн–пт).
- Навигация `← Пред. день / След. день →` всегда видна для ручного переключения.

**Шаг 4 — визуальный пример:**
```
Клиент: Магадан (UTC+10)  |  МП: Самара (UTC+4)
Разница: клиент опережает МП на 6 часов

Рабочие часы МП: 09:00–17:00 (UTC+4) = 15:00–23:00 (UTC+10)
Разумное время клиента: 09:00–20:00 (UTC+10)
Пересечение: 15:00–20:00 (UTC+10) = 09:00–14:00 (UTC+4)

Доступные слоты: 09:00 / 10:00 / 11:00 / 12:00 / 13:00 / 14:00 МП
             = 15:00 / 16:00 / 17:00 / 18:00 / 19:00 / 20:00 клиент
```
