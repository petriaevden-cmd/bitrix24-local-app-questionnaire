## Статус проекта

**Проект:** локальное приложение Битрикс24 для заполнения анкеты клиента в карточке лида.  
**Тип:** Local Application для Bitrix24 On-Premise.  
**Плейсмент:** `CRM_LEAD_DETAIL_TAB`.  
**Текущий статус:** аналитика и подготовка архитектуры / начальной структуры проекта.  
**Реализация:** начальный каркас проекта в репозитории создан, но бизнес-логика еще не доведена до рабочего состояния.  
**Текущая стадия:** уточнение требований и исправление пробелов исходного ТЗ.

---

## Кратко о проекте

Приложение должно открываться внутри карточки лида в Битрикс24 и позволять менеджеру КЦ:

- просматривать и заполнять анкету клиента;
- автоматически подтягивать уже сохраненные значения анкеты при повторном открытии;
- сохранять анкету в пользовательские поля лида `UF_CRM_KC_*`;
- дублировать результат в таймлайн лида;
- видеть расписание всех МП;
- бронировать слот клиента у выбранного МП;
- учитывать сложную логику часовых поясов клиента и МП;
- отдельно подсвечивать «горящие» слоты (`HOT`).

## Что уже подтверждено

Ниже перечислены решения, уже подтвержденные в переписке и считающиеся актуальными до отдельной отмены.

### 1. Повторное открытие анкеты

Если менеджер повторно открывает карточку лида, в которой анкета уже заполнялась, форма **не должна быть пустой**.

**Принятое решение:**
- при открытии анкеты подтягиваются уже заполненные значения из пользовательских полей лида;
- форма открывается в режиме просмотра/редактирования текущих значений;
- данные не обнуляются при повторном открытии.

### 2. Кто видит расписание менеджеров

**Принятое решение:**
- МКЦ видит расписание (календарь) **всех МП**;
- не одного выбранного менеджера, а общий срез по доступности менеджеров.

### 3. SALES_DEPT_ID

В исходном ТЗ было указано:

- `SALES_DEPT_ID = 42`

**Принятое решение:**
- на текущий момент это значение считается корректным и принимается как рабочее, пока не будет отдельного уточнения.

### 4. HOT-слот

Понятие `HOT` в исходном ТЗ было описано слабо.

**Подтвержденная бизнес-логика:**
- `HOT` — это «горящий слот»;
- это означает, что лид должен быть сейчас на встрече с МП, но пока не вышел на связь;
- такие слоты должны отдельно выделяться, потому что отмененных встреч быть не должно или по такому лиду нужно срочно дозвониться;
- такие лиды чаще всего обрабатываются МКЦ «прямо сейчас».

### 5. Поле официального дохода изменено

Исходное ТЗ содержало противоречие по полю `KC_INCOME_OFFICIAL`.

**Подтвержденное новое решение:**
вместо одного поля используется **два поля**:

1. `KC_INCOME_OFFICIAL` — тип `enumeration`, значения: `Да / Нет`
2. `KC_INCOME_OFFICIAL_AMOUNT` — тип `integer`, сумма официального дохода

**Дополнительное правило:**
- поле суммы официального дохода показывается только если в поле `KC_INCOME_OFFICIAL` выбрано значение `Да`.

**Следствие:**
- общее количество пользовательских полей увеличивается **с 26 до 27**.

### 6. Документирование всех изменений

**Принятое решение:**
- все дальнейшие изменения, уточнения, договоренности, доработки и расхождения с исходным ТЗ должны фиксироваться в `tz.md`.

### 7. Архитектура API-вызовов (proxy.php)

**Принятое решение:**
- первая рабочая версия строится через `BX24.js` в iframe Битрикс24;
- `proxy.php` остается как подготовленный резервный слой для серверных сценариев будущих версий;
- не завязываться на `proxy.php` в первой рабочей версии.
